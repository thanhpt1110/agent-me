# 2026-05-12 — Auto SFA

## User Request

Add an `Auto SFA` feature to agent-me that:

- exposes an Auto SFA button from Slack help,
- collects `username`, `devtest_folder_id`, `url_path`, `start`, and `finish`,
- updates `/localhome/local-thaphan/magic-auto/configs.json`,
- runs `uv run dtoperator.py sfa --task-owner "Your Name" -f` from the
  `magic-auto` repo, preserving the user-supplied owner name exactly as the
  subprocess argv value,
- streams terminal progress back into the Slack thread, and
- adds an equivalent dashboard tab with a form and live log output.

## Implementation

Shared runner:

- Added `agent_me.auto_sfa`.
- Parses keyed or ordered input.
- Validates `devtest_folder_id`, HTTPS/HTTP URL, and `yyyy-MM-dd` dates.
- Updates `magic-auto/configs.json`:
  - `devtest_folder_id`
  - `log_file_base_url`
  - `release_configs.source_code_path`
  - `release_configs.code_review_path`
  - every `release_configs` key containing `start`
  - every `release_configs` key containing `finish`
- Runs the `magic-auto` command with direct argv:
  `uv run dtoperator.py sfa --task-owner <username> -f`.
- Uses an advisory lock file in `magic-auto` so Slack and dashboard cannot
  concurrently clobber the shared config.

Slack:

- `/help` Block Kit includes an `Auto SFA` button.
- Plain `auto sfa`, `autosfa`, and `sfa` start the same flow.
- The flow stores active state in SQLite table `auto_sfa_flows`.
- The user can provide all values at once or continue with missing fields in
  the same thread.
- Once valid, the bridge starts the shared runner in the background and posts
  only newly produced terminal log lines into the same Slack thread.

Dashboard:

- Added `/auto-sfa`.
- Added `POST /api/auto-sfa/run`.
- Added `GET /api/sse/auto-sfa/{job_id}` for live terminal output.
- Added `AutoSFARunner` for process-local job state and SSE fan-out.

## Follow-up Fix

After Slack E2E, the first parser accepted only part of a complete form in
thread replies. The cause was brittle key-value extraction: inline follow-ups
such as `username: Thanh Phan, finish date: 2026-04-28` were parsed as one
username value, and Slack-formatted links like `<https://...|https://...>`
could fail URL validation.

Fix:

- Parse key-value pairs across the full Slack text, not only one key per line.
- Strip leading Slack bot mentions and plain `@agent-me`.
- Support comma/semicolon inline follow-ups.
- Normalize Slack auto-linked URLs back to the raw URL.
- Extract the `yyyy-MM-dd` date from date fields before validation.
- Reword Slack instructions so the user sees a friendly guided flow while the
  runner still receives deterministic fields.

Verification: `uv run ruff check .` and `uv run pytest` (`115 passed`).

## Slack Code Fence Follow-up

Slack E2E still reported missing `username` and `finish date` for a complete
form. The raw bridge log showed Slack wrapping the pasted form in a code block
with fence markers on the same lines as real values:

- opening fence on the same line as `username: Thanh Phan`
- closing fence on the same line as `finish: 2026-04-27`

The parser stripped those whole boundary lines, which removed exactly the first
and last fields. The fix now removes only the fence delimiters and preserves
line content. It also normalizes the bot mention before code-fence handling, so
`@agent-me` followed by a fenced form parses the same way.

Additional parser support:

- Accept `username la ...` / `username là ...` and `finish la ...` /
  `finish là ...` as friendly Vietnamese separators.
- Treat `Thanh Phan` as a real user-provided value, even when it matches the
  example text.

verification passed with `uv run ruff check .` and `uv run pytest`
(`117 passed`).

## 2026-05-13 Magic-Auto Flow Refresh

`magic-auto` was reimplemented as an HTTP-based DevTest operator. The agent-me
Auto SFA integration now follows that flow instead of the legacy 5-field
wrapper.

Changes:

- Use `uv run dtoperator.py sfa --user-login <login> -f`; the old
  `--task-owner` flag is no longer emitted.
- Keep the human-facing Slack/dashboard form compact with five required fields:
  `username`, `destination_folder_id`, `url_path`, `start`, and `end`.
- Map `username` (`thaphan`, or an email where only the local part is used) to
  the short DevTest login before spawning `magic-auto`.
- Treat `url_path` as the shared URL and reuse it for `log_file_base_url`,
  `source_code_path`, and `code_review_path`.
- Map `start` to all required start-date fields and `end` to all required
  finish-date fields. Defaults remain `devtest_project_id=1074`,
  `log_file_provider=Manual`, and `complexity_level=L2`; `source_folder_id`
  stays whatever is already present in `magic-auto/configs.json`.
- Update dashboard `/auto-sfa` to show the compact form and dark date inputs
  consistent with the rest of the dashboard.

Verification: focused Auto SFA/dashboard/Slack tests passed (`18 passed`).

## 2026-05-13 Magic-Auto Display Name + Task IDs Refresh

The latest local `magic-auto` README clarified that `dtoperator.py sfa -u` is
the DevTest `Automation Dev Linux` display name, not the short login. The
agent-me Auto SFA integration now follows that contract.

Changes:

- Replace the main Slack/dashboard `username` input with required
  `display_name`, for example `Thanh Phan`.
- Keep the compact required fields: `display_name`, `destination_folder_id`,
  `url_path`, `start`, and `end`.
- Add optional `source_folder_id`; the dashboard exposes a "use default"
  checkbox that disables the override field and preserves the existing config
  default.
- Add optional `task_ids`; when present the spawned command becomes
  `uv run dtoperator.py sfa -i <ids> --user-login <display_name> -f`, matching
  the magic-auto specific-ID flow.
- Keep the display-name mode as the default:
  `uv run dtoperator.py sfa --user-login <display_name> -f`.
- Add a dashboard-only credentials panel. The dashboard asks for custom
  DevTest `USERNAME` and `PASSWORD` by default; a "use default" checkbox hides
  the fields and falls back to host `.env` credentials. When supplied,
  agent-me injects the credentials into the child process environment for that
  run, excludes the password from public job JSON/SSE, and clears it from
  in-memory job history after the process exits. This makes `magic-auto`
  authenticate as the provided user so its cookie `loginid` maps Task Owner to
  the right account.
- Slack help now documents both modes and explicitly points personal
  credential overrides to the dashboard instead of asking users to paste
  passwords into Slack.

## 2026-05-13 Source Default + Trigger History

The dashboard source folder field no longer has a `use default` checkbox. It is
pre-filled with `50722` and includes a short note telling users to keep that
default unless they need another source folder. The dashboard sends that value
as the runtime `source_folder_id` override, so it does not depend on whatever
the last `magic-auto/configs.json` run left behind.

Auto SFA triggers are now persisted to SQLite in `auto_sfa_runs` with run id,
trigger time, display name, status, and source. The `/auto-sfa` page renders a
scrollable trigger history at the bottom and refreshes it after dashboard runs
start or finish. Slack Auto SFA runs write the same history table.

## 2026-05-13 Create/Release Split, Concurrency, and History Refresh

The Auto SFA dashboard was expanded from one release form into one Auto SFA
entry point with two distinct workflows.

Create SFA Tasks:

- This is the template-prep step before release.
- Required inputs are only `display_name` and `folder_id`.
- `template_ids` is optional for specific template IDs.
- `project_id` stays internal and fixed at `1072`; the UI does not show or
  submit it.
- The backend calls `magic-auto` `update-template`, filters by Template Owner
  display name when template IDs are not provided, and updates:
  `Automation Dev Linux = display_name`,
  `Automation Status Linux = In A&T`, and `Win_Linux = Linux_Only`.
- Create validation is isolated from release validation. Release-only errors
  such as missing `devtest_folder_id`, `url_path`, date fields,
  `source_code_path`, or `code_review_path` should only appear when running
  Release SFA Tasks.

Release SFA Tasks:

- Keeps the existing release flow and fields.
- The dashboard defaults `end` to the browser's current day and `start` to
  seven days before `end`.
- The runner now writes a per-job temporary magic-auto config and passes it
  with `-c`, instead of mutating the shared `magic-auto/configs.json`.

Concurrency and cancellation:

- The dashboard runner no longer enforces a single active Auto SFA job.
- Each run has its own job id, subprocess, SSE terminal stream, and temp config
  where needed.
- Credentials are injected into only that child process environment and are
  removed from in-memory public job state after the job exits.
- `Cancel operation` cancels only the selected job and terminates that job's
  subprocess; other users' jobs continue.
- `new terminal` resets only the current browser terminal attachment.

History:

- SQLite table `auto_sfa_runs` now has `flow_type` with values `create` or
  `release`.
- Existing rows are migrated with default `flow_type = release`.
- `/api/auto-sfa/history` returns both the combined rows and `runs_by_flow`.
- The dashboard removed `Recent Auto SFA jobs`; instead, the bottom trigger
  history follows the selected workflow tab and renders the same columns for
  each flow: `Run`, `Time`, `Display name`, and `Status`.

Verification:

- Focused dashboard/Auto SFA tests passed.
- Full verification passed with `uv run pytest -q` and `uv run ruff check .`.
